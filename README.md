# DTF Radar

DTF Radar is a lightweight competitor intelligence MVP for the DTF market.

It helps sellers, brands, and distributors monitor:
- pricing changes
- product launches
- promotional updates
- messaging shifts
- competitor activity trends

## Current status
This repository now includes a working backend-ready MVP skeleton with:
- Prisma data model
- Prisma init migration
- local PostgreSQL setup path
- competitor create/list/detail flows
- discovery suggestion + tracked page persistence
- alerts / change events data layer
- dashboard summary aggregation
- seed / demo data support
- production build/start verification
- Nginx / PM2 / systemd deployment guidance
- PM2 persistent run already validated on the current server

## Local / server setup

### 1. Install dependencies
```bash
npm install
```

### 2. Create environment file
```bash
cp .env.example .env
```

Example:
```bash
DATABASE_URL="postgresql://USER:PASSWORD@HOST:5432/dtf_radar?schema=public"
NEXT_PUBLIC_APP_URL="http://localhost:3000"
NODE_ENV="development"
```

### 3. Generate Prisma client
```bash
DATABASE_URL="postgresql://USER:PASSWORD@HOST:5432/dtf_radar?schema=public" npx prisma generate
```

### 4. Apply migrations
```bash
DATABASE_URL="postgresql://USER:PASSWORD@HOST:5432/dtf_radar?schema=public" npx prisma migrate dev
```

### 5. Seed demo data
```bash
npx tsx prisma/seed.ts
```

### 6. Start dev server
```bash
npm run dev -- --hostname 0.0.0.0 --port 3000
```

## Recommended production run flow
For a simple server deployment without Docker yet:

### Build
```bash
npm run build
```

### Start
```bash
npm run start
```

### Persistent run with PM2
```bash
pm2 start npm --name dtf-radar -- start
pm2 save
pm2 startup
```

By default Next.js serves on port `3000`.

For cloud-server access and longer-term deployment:
- use Nginx to proxy 80/443 to `127.0.0.1:3000`
- use PM2 or systemd to keep the process alive
- avoid exposing 3000 publicly long term

## Prisma 7 note
This project uses Prisma 7 with a PostgreSQL driver adapter for local direct connections.

That means:
- Prisma CLI uses `prisma.config.ts`
- runtime Prisma Client uses `@prisma/adapter-pg`
- `DATABASE_URL` must exist for CLI and runtime
- `seed.ts` explicitly loads `.env` via `dotenv/config`

## Current limitations
- discovery suggestions are rule-derived from `websiteUrl`, not real crawling yet
- change events are not generated by a crawler yet
- page snapshots / diff engine are not implemented yet
- auth / multi-workspace handling still uses a temporary default workspace id
- production process management is documented and PM2 is already running, but Nginx/domain/HTTPS are not yet configured
